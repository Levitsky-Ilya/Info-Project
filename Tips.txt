Code Style (Битнер)
	- Код чаще читается, чем пишется
	- Писать "хороший код" надо сразу
	- Потраченное время на "причесывание" кода всегда окупится со сторицей (whatever this is)
	- Документация
		- Комментарии фиксированного формата
		- Разумный подход в коментировании кода
	- Декомпозиция - практика разбиения кода на более мелкие части
		- Предпочтительна декомпозиция посредством нисходящего проектирования (от общих черт к подробностям)
		- Если все стало слишком запутанным, то стоит сделать рефакторинг
	- Присвоение имен
		- Хорошее имя - имя, которое говорит само за себя (Self-meaning)
		- Венгерская нотация (причем сначала глагол, потом существительное)
		- Единый стиль имен для всех участников проекта
	- Корректное использование языка: никаких "магических чисел в коде"
	- Форматирование
	
	В начале файла:
		- Авторство или копирайт
		- Имя файла
		- Назначение файла
		- Описание алгоритма
		- Changelog
	После include'ов, в !каждой! функции:
		- Описание входящих аргументов
		- Описание возвращаемых значений
		- Описание ошибок: какие exeption'ы порождаются
		- Описание алгоритма
	

Комментарии
	- Зачем внесены изменения
	- Что именно было изменено
	- На что изменили, как теперь будет работать этот участок
	
Использовать const, static
	Он [static] значит для компилятора, что эта функция работает только внутри данного модуля, например этого файла с исходным кодом. (Илья)

Форматирование (.md файлы)
	- "<>" делает текст внутри себя заголовком
	- "=======" (вроде как 7 и больше, надо проверить) на следующей строке после текста делает его заголовком
	- "<<>>" выдаст "<>", не знаю, почему
